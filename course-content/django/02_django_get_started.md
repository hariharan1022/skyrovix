# 2. Django Get Started

## 📘 Introduction
This module covers installing Django, creating your first project, understanding the project structure, and running the development server. By the end, you'll have a working Django project and understand the role of each file generated by the `startproject` command.

## 🧠 Key Concepts
- **pip install django**: The standard way to install Django from PyPI into your Python environment.
- **django-admin**: Django's command-line utility for administrative tasks — creating projects, checking configuration, and managing apps.
- **manage.py**: A project-specific wrapper around `django-admin` that automatically uses your project's settings.
- **Project Structure**: Django generates a package with `settings.py` (configuration), `urls.py` (URL declarations), `wsgi.py` (WSGI server entry), `asgi.py` (ASGI server entry), and `manage.py`.
- **Development Server**: Django's lightweight built-in server (`runserver`) for development — automatically reloads on code changes.

## 💻 Syntax
```bash
# Install Django
pip install django

# Verify installation
python -m django --version

# Create a new project
django-admin startproject mysite

# Navigate into project directory
cd mysite

# Run development server
python manage.py runserver

# Run on a different port
python manage.py runserver 8080

# Run on a specific IP/port
python manage.py runserver 0.0.0.0:8000
```

## ✅ Example 1 - Basic: Installing Django and Creating Your First Project

**Problem:** Install Django and create a new project called `myblog`.

**Code:**
```bash
# Create and activate a virtual environment
python -m venv venv
.\venv\Scripts\activate   # Windows
# source venv/bin/activate  # macOS/Linux

# Install Django
pip install django

# Verify installation
python -m django --version
# Output: 5.1

# Create project
django-admin startproject myblog

# View project structure
cd myblog
dir
```

**Output/Description:**
```
Directory: ...\myblog

Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
d-----        23-06-2026     14:35                myblog
-a----        23-06-2026     14:35            690 manage.py
```

**Explanation:** `pip install django` downloads and installs the latest Django version from PyPI. `django-admin startproject myblog` creates a `myblog/` directory containing `manage.py` and an inner `myblog/` package with settings files. The outer directory is the project container; the inner directory is the Python package for your project.

## 🚀 Example 2 - Intermediate: Exploring Project Structure and Running the Server

**Problem:** Examine the files generated by `startproject`, understand their purpose, and start the development server.

**Code:**
```bash
# After creating the project with: django-admin startproject myblog
# List all project files including the inner package
cd myblog
Get-ChildItem -Recurse -Name
```

```bash
# Run the development server
python manage.py runserver
```

```
# Project structure:
myblog/
├── manage.py              # Command-line utility for this project
└── myblog/                # Python package for project settings
    ├── __init__.py         # Makes this directory a Python package
    ├── asgi.py             # ASGI configuration for async servers
    ├── settings.py         # ALL project configuration
    ├── urls.py             # URL declarations (like a table of contents)
    └── wsgi.py             # WSGI configuration for production deployment
```

**Output/Description:** The server starts at `http://127.0.0.1:8000/` showing Django's default "Congratulations!" page. Files breakdown:
- `manage.py` — same as `django-admin` but uses project-specific settings
- `settings.py` — database config, installed apps, middleware, templates, static files
- `urls.py` — maps URLs to view functions
- `wsgi.py` — entry point for WSGI-compatible servers (Gunicorn, uWSGI)
- `asgi.py` — entry point for ASGI-compatible servers (Daphne, Uvicorn) for async support

**Explanation:** The development server (`runserver`) auto-reloads when you save Python files — no manual restart needed. It's designed for development only, not production. The warning about unapplied migrations is normal for a fresh project.

## 🏢 Real World Use Case
A freelance developer starts a new client project — a restaurant reservation system. They create a virtual environment, install Django, and run `django-admin startproject restaurant_project`. The project structure provides an organized foundation. The developer runs `python manage.py runserver` to immediately see the project is working. The auto-reload feature saves hours during development since every code change is reflected instantly. The developer later deploys using Gunicorn (WSGI) for production, using the `wsgi.py` file as the entry point.

## 🎯 Interview Questions

**Q1: What is the difference between `django-admin` and `manage.py`?**
**A:** `django-admin` is Django's global command-line tool that can be used with any Django project. `manage.py` is a project-specific wrapper automatically created by `startproject` that sets the `DJANGO_SETTINGS_MODULE` environment variable to the project's settings module. `manage.py` is more convenient for project tasks because it already knows which settings to use.

**Q2: What is the purpose of `__init__.py` in the project folder?**
**A:** In Python, `__init__.py` marks a directory as a Python package, allowing the directory to be imported as a module. Django's project structure uses this so that settings, URLs, and other modules can be imported (e.g., `from myblog.settings import ...`).

**Q3: Why shouldn't `runserver` be used in production?**
**A:** Django's development server is single-threaded, not security-hardened, and not performance-optimized. It cannot handle concurrent requests efficiently and lacks production-grade features like request buffering, load balancing, and proper error handling. Use WSGI servers like Gunicorn, uWSGI, or ASGI servers like Daphne for production.

**Q4: What does the "You have 18 unapplied migration(s)" warning mean when you first run the server?**
**A:** Django comes with built-in apps (admin, auth, contenttypes, sessions) that have database tables. The warning means these tables haven't been created yet. Running `python manage.py migrate` creates the necessary database tables.

**Q5: What is WSGI and ASGI in the context of Django?**
**A:** WSGI (Web Server Gateway Interface) is a synchronous protocol for Python web servers to communicate with applications. ASGI (Asynchronous Server Gateway Interface) is the asynchronous successor supporting both sync and async code. Django provides both `wsgi.py` and `asgi.py` as entry points for deployment with respective server types.

## ⚠ Common Errors / Mistakes
- **Installing Django globally without a virtual environment**: Causes version conflicts across projects and requires system-level permissions.
- **Forgetting to activate the virtual environment**: Running `django-admin` or `python manage.py` without the venv active uses the system Python.
- **Running `django-admin startproject` inside an existing project**: Creates nested projects, causing import errors and confusion.
- **Confusing the outer and inner project directories**: The outer directory is a container; the inner directory (same name as project) is the Python package.
- **Binding to `0.0.0.0` without realizing it's accessible on the network**: Poses security risks during development.
