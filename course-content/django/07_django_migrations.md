# 7. Django Migrations

## 📘 Introduction
Django migrations are the system for propagating changes made to models (adding fields, deleting models, etc.) into the database schema. Migrations are Python files that describe the database changes to apply. They enable version control for your database schema, team collaboration, and the ability to roll back changes.

## 🧠 Key Concepts
- **makemigrations**: Analyzes your models and creates new migration files capturing the changes needed.
- **migrate**: Applies pending migrations to the database.
- **sqlmigrate**: Shows the SQL that a migration would execute — useful for reviewing before applying.
- **showmigrations**: Lists all migrations and their applied/unapplied status.
- **Migration Files**: Python files in `app/migrations/`, numbered sequentially (e.g., `0001_initial.py`, `0002_auto_...`).
- **Initial Migrations**: The first migration for an app that creates all initial tables.
- **Rolling Back**: Use `python manage.py migrate app_name migration_number` to revert to a specific migration.
- **Data Migrations**: Migrations that manipulate existing data (using `RunPython`).
- **Migration Dependencies**: Migrations specify dependencies on previous migrations in the same app or migrations in other apps.

## 💻 Syntax
```bash
# Create migrations for all apps
python manage.py makemigrations

# Create migrations for a specific app
python manage.py makemigrations blog

# Apply all pending migrations
python manage.py migrate

# Apply migrations for a specific app
python manage.py migrate blog

# Show the SQL for a migration
python manage.py sqlmigrate blog 0001

# List all migrations and their status
python manage.py showmigrations

# Rollback to a specific migration
python manage.py migrate blog 0001

# Rollback all migrations for an app (unapply everything)
python manage.py migrate blog zero
```

```python
# Example migration file (generated by makemigrations)
# blog/migrations/0001_initial.py
from django.db import migrations, models

class Migration(migrations.Migration):
    initial = True
    dependencies = []
    operations = [
        migrations.CreateModel(
            name='Article',
            fields=[
                ('id', models.BigAutoField(...)),
                ('title', models.CharField(max_length=200)),
                ('content', models.TextField()),
            ],
        ),
    ]
```

## ✅ Example 1 - Basic: Creating and Applying Initial Migrations

**Problem:** Create a `Category` model and migrate it to the database.

**Code:**
```python
# models.py
from django.db import models

class Category(models.Model):
    name = models.CharField(max_length=100)
    slug = models.SlugField(unique=True)
    created_at = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return self.name
```

```bash
# Step 1: Create migration file
python manage.py makemigrations
# Output: Migrations for 'blog':
#   blog/migrations/0001_initial.py
#     - Create model Category

# Step 2: Preview the SQL
python manage.py sqlmigrate blog 0001
```

```sql
-- SQL that will be executed:
BEGIN;
--
-- Create model Category
--
CREATE TABLE "blog_category" (
    "id" integer NOT NULL PRIMARY KEY AUTOINCREMENT,
    "name" varchar(100) NOT NULL,
    "slug" varchar(50) NOT NULL UNIQUE,
    "created_at" datetime NOT NULL
);
COMMIT;
```

```bash
# Step 3: Apply to database
python manage.py migrate
# Output: Applying blog.0001_initial... OK
```

**Output/Description:** The `blog_category` table is created in the database. Running `showmigrations` shows `[X] 0001_initial` (marked applied). The migration file is portable — it can be applied on any database backend.

**Explanation:** `makemigrations` compares current models against the last migration state and generates Python files describing the delta. `migrate` reads the migration files and executes SQL in order. Each migration tracks its `dependencies` to ensure correct ordering. The SQL varies by database backend (SQLite above; different for PostgreSQL/MySQL).

## 🚀 Example 2 - Intermediate: Making Model Changes and Data Migrations

**Problem:** Add a `priority` field to the `Category` model, then populate it with default values using a data migration.

**Code:**
```python
# models.py — After modification
from django.db import models

class Category(models.Model):
    name = models.CharField(max_length=100)
    slug = models.SlugField(unique=True)
    priority = models.IntegerField(default=0)    # New field
    created_at = models.DateTimeField(auto_now_add=True)
```

```bash
# Create migration for the new field
python manage.py makemigrations
# Output: blog/migrations/0002_category_priority.py
```

```python
# Auto-generated migration file
# blog/migrations/0002_category_priority.py
from django.db import migrations, models

class Migration(migrations.Migration):
    dependencies = [
        ('blog', '0001_initial'),
    ]
    operations = [
        migrations.AddField(
            model_name='category',
            name='priority',
            field=models.IntegerField(default=0),
        ),
    ]
```

```bash
# Create a data migration to assign initial priorities
python manage.py makemigrations blog --empty --name assign_priorities
# Output: blog/migrations/0003_assign_priorities.py
```

```python
# blog/migrations/0003_assign_priorities.py — Edited data migration
from django.db import migrations

def assign_priorities(apps, schema_editor):
    Category = apps.get_model('blog', 'Category')
    categories = Category.objects.all()
    for i, category in enumerate(categories):
        category.priority = (i % 3) + 1  # Cycle 1, 2, 3
        category.save()

def reverse_priorities(apps, schema_editor):
    Category = apps.get_model('blog', 'Category')
    Category.objects.all().update(priority=0)

class Migration(migrations.Migration):
    dependencies = [
        ('blog', '0002_category_priority'),
    ]
    operations = [
        migrations.RunPython(assign_priorities, reverse_priorities),
    ]
```

```bash
# Apply the schema and data migrations
python manage.py migrate
```

**Output/Description:** The `priority` column is added to the `blog_category` table. Then the data migration runs a Python function that assigns a priority (1, 2, or 3) to each existing category. The migration can be rolled back to revert priorities to 0.

**Explanation:** Data migrations use `RunPython` with two callables: forward function and reverse function. Use `apps.get_model()` (not model imports) so the migration uses the correct historical model state. Always provide a reverse function for rolling back. Data migrations are regular migration files that can depend on schema migrations.

## 🏢 Real World Use Case
A development team with 5 developers works on a Django SaaS product. Each developer works on a different feature branch. When a developer adds a model field, they run `makemigrations` and commit the migration file. On deployment, `migrate` runs all pending migrations in order. If a buggy migration is deployed, `migrate app prev_migration` rolls back. For a data migration that backfills user profile data from a legacy system, `RunPython` processes 50,000 records. The migration is reversible for rollback scenarios.

## 🎯 Interview Questions

**Q1: How does Django detect model changes and create migrations?**
**A:** Django stores the current state of all models in the migration files. When you run `makemigrations`, Django compares your current `models.py` definitions against the last recorded state in the migration files. The difference is encoded as migration operations (CreateModel, AddField, RemoveField, AlterField, etc.) in a new migration file.

**Q2: What is `sqlmigrate` and why would you use it?**
**A:** `sqlmigrate` outputs the actual SQL statements a migration would execute without applying them to the database. It's useful for: reviewing changes before production deployment, understanding what SQL Django generates, debugging migration issues, and sharing SQL with database administrators who need to approve schema changes.

**Q3: How do you handle conflicts when two developers create migrations with the same number?**
**A:** When pulling changes, if two branches have migrations with the same number but different content, you'll get migration conflicts. Resolve by: (1) run `makemigrations --merge` to create a merge migration that depends on both conflicting migrations, or (2) manually edit migration dependencies to create a linear chain.

**Q4: What is the difference between `makemigrations` and `migrate`?**
**A:** `makemigrations` is the planning phase — it creates migration files (Python code describing schema changes) based on model changes. `migrate` is the execution phase — it reads migration files and applies the actual SQL to the database. `makemigrations` does NOT touch the database; `migrate` does.

**Q5: How do you create a migration that adds a non-nullable field without a default to an existing table?**
**A:** Django handles this in two migrations. First, `makemigrations` creates a migration that adds the field with `null=True`. Then, a data migration populates the field for existing rows. Finally, another migration alters the field to set `null=False`. This is because adding a non-nullable field requires existing rows to have a value.

## ⚠ Common Errors / Mistakes
- **Editing migration files manually after they've been applied**: Causes checksum mismatches and broken migration chains. Always create a new migration instead.
- **Running `migrate` before `makemigrations`**: Nothing happens — `migrate` only processes existing migration files. Always run `makemigrations` first.
- **Forgetting to run `makemigrations` after model changes**: Changes exist in Python but not in database — queries fail with "no such column" errors.
- **Committing migration files without running them**: Other developers will get unapplied migrations and might forget to run them.
- **Using model imports in data migrations instead of `apps.get_model()`**: Causes migration failures when models change in later migrations.
- **Not providing a reverse function for `RunPython`**: Rolling back through a data migration without a reverse function raises `IrreversibleError`.
